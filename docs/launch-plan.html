Launch Strategy · v2 · 2026-07-19

Cookrew 统一品牌宣发计划

决策已定（2026-07-19）：Cava 整体并入 COOKREW 品牌——Cookrew 立项更早、 域名 cookrew.dev 已在手，且合并直接消解了 "cava" 与 karlstav/cava（终端音频可视化）的撞名风险。 本计划覆盖：改名执行、产品架构、四阶段宣发、渠道与指标。

01合并后的产品架构：一个品牌，两个形态

COOKREW DESKTOP（原 Cava）

Electron + React · MIT 开源 · 本地优先 · 获客入口

  • 无限画布：终端、便签、浏览器节点用绳缆连线，连线即授权
  • cookrew CLI 是完整 agent API（ask / note / browser / recruit / routine）
  • 语音模式 + 手机伴侣（扫码即用，免安装）
  • 对标 Maestri（原生 Swift、仅 macOS）——差异化是开源 + 跨平台

COOKREW CLOUD（原 Cookrew Web）

Next.js Web · KrewHub 后端 · 多人实时 · 商业化层

  • Arcade 实时编排画布，master-workers 模型
  • Desktop 用户通过 cookrew bridge 把本地画布接入云端房间
  • 团队空间 / 云执行 / 订阅制落点
一句话定位（对外口径）：
COOKREW — "The multiplayer kitchen for humans & AI agents."
DESKTOP 开源本地厨房："Your agents, on one canvas — open-source, cross-platform."
CLOUD 多人层："When your solo kitchen goes multiplayer." 厨房叙事（下发目标 → 召唤队伍 → 看它们烹饪）沿用既有落地页文案，一以贯之。

单品牌的直接红利：宣发期不再需要隐藏云端叙事——cookrew.dev 从发布第一天 就承载 Desktop 下载 + Cloud waitlist，开源流量从 day 1 开始为商业化层蓄水。

repo 尚未公开、无历史包袱，做干净切换，不留兼容别名：

项现值改为备注

环境变量CAVA_SOCKET / CAVA_TERMINAL_ID / CAVA_CLICOOKREW_SOCKET / COOKREW_TERMINAL_ID / COOKREW_CLIagent 提示词、README 同步更新

执行顺序：先全局改名（代码里的 cava 字符串、CLI 文件名、socket 路径、测试）， 再首次 git 提交——让 repo 历史从第一天就是干净的 Cookrew，公开后不存在 "改名前考古"。

03发布前另外两件必办事

  • Maestri 关系措辞：全部物料统一 "inspired by Maestri"，不用 clone/replica；不使用对方商标截图；发布前给 Maestri 作者发一封友好邮件打招呼——HN 帖子里原作者现身是常事，善意关系是加分项，敌意是公关事故。
  • 可安装性：宣发日必须有开箱即用产物——GitHub Releases 三平台安装包（electron-builder：dmg / AppImage / exe）+ CI 自动构建。"clone 下来自己 build" 会损失 80% 的尝鲜用户。

04宣发路线图（四个阶段）

PHASE 0 · 2026-07-20 → 08-02（两周）

改名 + 发布准备

  • 发布周每天值守 issue 和评论区；当天修 bug 当天发 patch release，这本身就是最好的宣传

PHASE 3 · 2026-Q4 · Cloud 上线

"Your solo kitchen goes multiplayer"

  1. 协议先行：把 cookrew CLI 的 ndjson socket 协议整理成公开 spec；KrewHub 实现同一协议——结合的技术地基，也是一篇好的技术博客
  2. cookrew bridge：本地画布一条命令接入 Cloud 房间，本地终端节点实时出现在云端 Arcade，队友在浏览器里围观/下发任务
  3. 工作区互通：workspace.json ↔ Cloud bundle 双向导入导出
  4. Beta 招募：waitlist（Phase 0 起蓄水）+ Discord 社区定向邀请；云端房间/团队空间是订阅制商业化落点

07风险清单

  • 命名撞车：已随品牌合并消解（cookrew 无同名知名项目；crew 别名暂缓，避开 chromebrew）。
  • 改名残留：合并后代码/文档/提示词里残存 "cava" 字样会显得半成品。公开前全局 grep 清零，CI 加一条防回归检查。
  • Electron 质疑：HN 必然出现。对策：README/FAQ 预埋回答（跨平台是产品前提，Maestri 已占 macOS 原生位），首评主动提及，不辩解、给理由。
  • Maestri 作者反应：提前打招呼 + 措辞规范（第 3 节）；若对方公开不满，回应基调是致敬与感谢，绝不对线。
  • Desktop/Cloud 预期管理：单品牌意味着用户会从 day 1 问 "Cloud 呢"。waitlist + 公开 roadmap 管理预期，别承诺具体日期。
  • 安全面暴露：browser evaluate / recruit / socket 是攻击面，开源后会被人盯。发布前跑一轮安全审计，写 SECURITY.md，出问题快速响应。
