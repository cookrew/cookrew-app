Cookrew · mobile · design proposal

The composer should ride the keyboard, not hide behind it.

Redesigning the phone input: the message field and its send & arrow controls lift with the keyboard and float over the terminal — one clean pill instead of a chrome-heavy bar wedged under it.

Design first. Nothing below is built yet — toggle the phone to feel the interaction, pick a layout, then we implement.

01What breaks today

On a phone the layout viewport stays full height while the keyboard slides up over it. The composer sits in normal flow at the bottom of the popout — so the keyboard lands on top of it.

✕Typed blind

The input and send button are covered by the keyboard the moment you focus the field — you can't see what you're writing.

✕Never lifts

The mobile companion doesn't watch visualViewport at all, so nothing reacts to the keyboard opening.

✕Five-control bar

attach · mic · field · speak · send, boxed in hard cream chrome — cramped and busy on a narrow screen.

02The fix, live

The composer leaves the layout and becomes a floating pill pinned above the keyboard, over the transcript. Flip the keyboard and swap layouts on the real mock.

Rule 1 · follow

Pinned above the keyboard

The pill's bottom is driven by the live keyboard inset, so it rests just over the top keyboard row and glides with it — open, close, or a mid-drag.

Rule 2 · float

Over the terminal, not under it

It leaves layout flow and floats in front of the black transcript. A soft fade behind it keeps the last lines legible; the transcript keeps its full height.

Rule 3 · simplify

Field first, one send

The message field is the hero; the amber send arrow is the only always-lit control. Attach · mic · speak and the terminal arrow-keys move one tap away.

03Three layouts

Option A · recommended

One pill, tools tucked

A single rounded pill: ＋ opens attach/mic/speak, ⌨ pops the arrow-key rail, amber ➤ sends. The calmest bar — everything secondary is a tap away.

Option B

Pill + always-on key rail

The arrow & Esc rail floats permanently just above the pill. Terminal control is never hidden — costs a little height and a busier resting state.

Option C

Minimal, expands on focus

Collapsed it's just field + send. Focus (keyboard up) expands it upward to reveal tools and the key rail — most minimal at rest, one more state to reason about.

04How it's wired

The main app already publishes a live keyboard inset; the companion just needs the same listener. The composer positions off it.

/* the companion gains the listener the main app already has */
visualViewport.addEventListener('resize', () => {
  const kb = Math.max(0, innerHeight - visualViewport.height - visualViewport.offsetTop)
  root.style.setProperty('--kb-inset', kb + 'px')   // keyboardInset() is already unit-tested
})

/* the composer floats above it, over the transcript */
#composer {
  position: fixed;                          /* out of flow → in front of the terminal */
  left: 10px; right: 10px; z-index: 40;
  bottom: calc(var(--kb-inset, 0px) + env(safe-area-inset-bottom) + 10px);
  transition: bottom .24s cubic-bezier(.22,1,.36,1);
}

Follow

Bottom offset = --kb-inset + safe-area. Reuses keyboard-inset.ts (already tested) in the app; adds the same listener to client.html.

Float

position:fixed over the transcript; a bottom fade on #popBody keeps trailing lines readable. No layout reflow, no lost height.

Reduced motion

The glide honors prefers-reduced-motion — the pill snaps instead of sliding.

Reach

Send & the popped controls keep a 44px touch target; the pill clears the home-indicator via safe-area.

05If Option A is a go

  1. Add the visualViewport listener to mobile/client.html (mirrors keyboard-inset.ts) so --kb-inset is live on the companion.
  2. Float #composer — fixed, keyboard-offset bottom, over the transcript — and restyle it as the pill; add the ＋ tools popover and ⌨ key-rail.
  3. Mirror the pill in the app's Dock / VoiceBar phone view so both surfaces match.
  4. QA on a real device: open/close keyboard, mid-drag, safe-area, reduced-motion.
