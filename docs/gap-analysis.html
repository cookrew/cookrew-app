Agent Canvas Showdown

Three spatial "canvas of AI agents" apps, evaluated hands-on by three parallel Opus scouts — each one deeply using its target through its real UI, CLI, docs, and app bundle. Here's what each does, and exactly where the open-source clone (Cookrew) stands against the two commercial products.

AgentGrid2.7.9
The most product-complete: multi-provider orchestration with voice, mobile, cloud sync and billing.
stack: Electron · React · TipTap · Monaco · xterm/node-pty · Claude Agent SDK + MCP · express/ws · Better Auth
▸ Broadest surface, cross-platform + iOS
Maestri0.34.3
The most elegant & deep: a native, on-device, Apple-platform craft object.
stack: Native Swift/SwiftUI/AppKit · SwiftTerm+Metal · WebKit portals · Apple FoundationModels · APFS floors · Sparkle
▸ Deepest isolation & privacy — macOS only
Cookrew0.1.0
The open-source clone: faithful core loop, cross-platform, MVP-grade.
stack: Electron · React Flow · xterm/node-pty · unix-socket CLI (no SDK/MCP) · webview portals · MIT
▸ Real & working, ~60% of the way there

All three share the same thesis — terminals, notes and browsers as free-floating panes on an infinite canvas, with AI agents orchestrated across them — but they diverge sharply on architecture, isolation, remoteness, and how agents are wired together.

Where Cookrew falls short — ranked gaps

Cookrew vs Maestri

MissingSeverityDetail
Floors (isolation)HIGHMaestri gives each agent an APFS copy-on-write clone on its own git branch, mirrored back to origin. Cookrew is a single flat workspace — recruit --dir only sets a cwd. No parallel-agent isolation. Cross-platform path: git worktrees (README-endorsed, unbuilt).
Environments (remote)HIGHMaestri bridges to SSH / Docker / Sandbox / custom runtimes over a Bridge Port with per-terminal tokens. Cookrew is local-unix-socket only — no remote or containerized terminals.
RolesMEDThe data model has a role field and recruit --role accepts it, but nothing consumes it — no role library, no prompt injection, no --replace hot-swap. A cheap win: the plumbing is already there.
Batuta (command palette)MEDNo fuzzy palette, no cross-node search, no keyboard-driven navigation across the canvas.
File-tree / editor / git paneMEDNo filesystem browser, no in-canvas code editor, no git diff/graph view.
Prompt composerMEDNo @mention composer, no image-as-native-pixels; prompts are raw strings via ask or the mobile textbox.
Ombro (local AI supervisor)MEDNo passive on-device companion watching agents and summarizing.
Groups / tidy / native polishLOWNo node grouping or auto-arrange; xterm.js vs SwiftTerm+Metal; no Sparkle auto-update.

Cookrew vs AgentGrid

MissingSeverityDetail
Cross-device / cloud syncHIGHAgentGrid mirrors panes/terminals/notes/layout two-way across paired machines + a native iOS app. Cookrew's mobile piece is a thin LAN remote into one running desktop — no account, no cloud state, nothing if the Mac is off.
Background daemonHIGHAgentGrid can own PTYs/workers independently of the window — close or crash the GUI without losing agents. Cookrew's agents die with the app.
MCP orchestration + spawn_roleMEDAgentGrid's permissioned canvas-blind master/worker model over MCP with a project-committed .agent-grid/roles.json contract. Cookrew has recruit but no role contract or SDK workers.
SDK/PTY cost leverMEDAgentGrid's SDK backend runs workers without consuming the Max subscription quota — a real cost dial Cookrew lacks.
Account / billing / iOS appMEDEncrypted secrets, Better Auth + Google sign-in, in-app usage/limits, native iOS TestFlight client.
Voice on non-macOSMEDAgentGrid's voice is cloud realtime (any OS). Cookrew's spoken output is hardcoded to macOS say.
⚠ Security finding (Cookrew, flagged by the scout): the mobile server has no authentication and CORS: * — any device on the LAN can POST to /api/terminal/:id/input and inject shell commands into any PTY. It ships a self-signed cert but no per-terminal tokens (the exact mechanism Maestri's environment bridge uses). This is a real remote-code-execution surface on an untrusted network and should gate any multi-user or public-Wi-Fi use.

What each does best — the real moats

AgentGrid

  • Realtime voice + call history — telephony-style agent channel, multilingual
  • Two-way cross-device sync + native iOS client
  • Background daemon — work survives GUI crash
  • SDK vs PTY cost lever — skip Max quota
  • Config-as-contract — committable .agent-grid/, notes-inbox polling

Maestri

  • APFS floors — instant, cheap, git-branch isolation, IDE-visible
  • Ombro — on-device Apple FM supervisor, fully private
  • Environments — SSH/Docker/Sandbox parity with token auth
  • Batuta — deterministic cross-workspace command palette
  • Native craft — Metal SwiftTerm, WebKit, connection physics

Cookrew

  • Open source (MIT) — the only inspectable/forkable one
  • Cross-platform — Electron reaches win/linux; Maestri can't
  • Faithful core loop — quiescence ask, connectivity=auth, Maestro-gating all real
  • Voice + mobile — reaches past Maestri toward AgentGrid
  • Tiny, readable — ~10 modules, easy to extend

The verdict — and Cookrew's path forward

Cookrew is a genuinely working MVP (~6/10 maturity) that faithfully clones Maestri's single-machine core loop and even overshoots it on voice + mobile (borrowed from AgentGrid). The hard parts — real PTYs, a headless mirror, quiescence-based ask, socket CLI, verified portal automation — are done and driven end-to-end by a real Claude agent on the canvas. What separates a "canvas of terminals" from a product is the remaining, differentiating ~40%: isolation, remote environments, roles, and sync.

The scouts' shared recommendation, in priority order:

  1. Floors via git worktrees — the biggest Maestri gap, cross-platform, already README-endorsed. Turns Cookrew from single-flat into real parallel-agent isolation.
  2. Wire up roles — the model and --role flag already exist; add a role library + prompt injection + --replace. Cheapest high-value win.
  3. Auth on the mobile/socket surface — per-terminal tokens (Maestri's own design) close the RCE hole and unblock everything remote/multi-user.
  4. Environments bridge — the remote-TCP + token architecture is already documented from the reverse-engineering; build SSH/Docker terminals on it.
Method: three Claude Opus 4.8 scouts recruited onto the Maestri canvas via maestri recruit, each deeply using one product hands-on (live CLI, app-bundle teardown with otool/asar strings, screencapture of running UIs, live portal driving) and writing a structured report.
