Defects

HIGHNote/Browser placement silently broken on the companion

crypto.randomUUID() only exists in secure contexts; the companion is served over plain HTTP on a LAN IP, so it is undefined. App.tsx:258 (note) and App.tsx:271 (browser) call it to mint the node id → unhandled rejection, no node created, no user feedback, and the tool stays stuck in NOTE/BROWSER mode because setTool('select') sits after the awaited addNode.

Repro: open the companion, tap ✎ NOTE, tap the canvas. Terminal placement uses server-side creation and is likely unaffected.

HIGHZoomed card stacks multiple fullscreen overlays — the wrong agent's terminal is interactive

zoom-lod.ts activates a card's full overlay when it covers ≥80% of the stage in either dimension, so partially visible neighbors qualify too. In remote mode (zoom-lod.ts:94-98) every active rect is forced to the whole stage → all active overlays render fullscreen, stacked; the interactive one is whichever is last in DOM order, not the card the user clicked.

Repro: click the "Magpie" card → 3 .lod-overlay elements mount at z-index 30; the composer resolves to a different agent's overlay. Anything typed goes to the wrong agent's terminal.

HIGHAll API calls hang while a card is zoomed open

With 3 overlays mounted, their streaming PTY-attach connections plus the state stream exhaust the browser's six-connections-per-origin HTTP/1.1 pool, so every subsequent fetch queues indefinitely. TurnHistoryPanel's listTurns never resolves → "Loading turns…" forever.

Evidence: in-page fetch('/api/terminal/test-id/turns') pending >6s while zoomed; resolved instantly once overlays unmounted. Same endpoint via curl: 200 in 37 ms. Consequence of Defect 2, but the connection budget is worth fixing independently.

MEDIUMCLI: cookrew browser screenshot advertised but not implemented

cookrew help lists screenshot; invoking it returns Unknown browser command 'screenshot'. Blocked visual-evidence capture for this whole pass.

MEDIUMCLI: cookrew browser fill silently fails on React-controlled inputs

fill reported "Filled" but the value stayed empty (React resets a bare .value assignment). Setting the value via the native HTMLInputElement value setter and dispatching an input event works — the CLI should do that.

Lower-confidence observations

  • ◀ pager on a 1-turn card is enabled but does nothing when clicked (minor polish, or an intended LIVE→turn-1 toggle whose rendering happened to be identical).
  • Both companion pages hard-reloaded at least once mid-session (injected globals wiped, canvas reset). Cause undetermined — possibly desktop-side webview lifecycle.
  • Placement failures give zero user feedback — worth a toast/error UI of its own beyond Defect 1.
  • The header "MOBILE" chip looks like a tab but is a static badge (Header.tsx:26) — invites dead taps on a phone.
  • CLI evaluate runs scripts in a shared global scope (repeat const names throw) and errors on non-string returns — rough edges for tooling users.
  • Node close ✕ buttons render at ~2×3 px at overview zoom — unusable tap targets on a real phone.

Coverage gaps

  • True phone-width layout: untestable — both views 1100×711, no viewport resize in the CLI.
  • Terminal placement, actual fork, actual file attach, connect-cable completion: intentionally not executed (would mutate the live workspace / spawn processes / open native dialogs).
  • Canvas drag-pan and node drag: no drag primitive in the CLI.
