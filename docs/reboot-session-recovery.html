The reboot that forgot the agents

After a system reboot, every Cookrew Dev agent came back with a blank memory. Their real conversations were never lost — Cookrew was resuming a session id that had quietly diverged from the one Claude actually writes.

Fixed in source · 287 tests green · recovery armed

Root cause · the phantom id

Cookrew binds every Claude terminal to a session id so it can find that terminal's transcript on disk. The chosen id is decided at spawn in spawnTracked():

// src/main/index.ts — before
const stored = t.claudeSessionId && UUID_RE.test(t.claudeSessionId) ? t.claudeSessionId : null
const sessionId = stored ?? extractSessionFlag(command) ?? randomUUID()   // ← mints a fresh id
effective = claudeSpawnCommand(command, t.cwd, sessionId)

That fresh id is only ever honored if Claude actually boots with it. But Cookrew runs each terminal inside a persistent tmux session, and tmux's new-session -A reattaches a live session and ignores the boot command. So when the id was minted for a terminal whose Claude was already running under its own self-generated id, the new id never reached Claude — it was written to workspace.json and diverged silently.

A second bug compounded it: SessionTurnSync was watching <phantom-id>.jsonl — a file that never existed — so each terminal's turn history froze at the pre-divergence scrape. That's why matching turn history alone can't reliably find the real session.

The fix

A new resolveClaudeSessionId() replaces the blind randomUUID(). When a stored id has no session file, it recovers the terminal's real session from turn history — newest on ties — instead of booting blank. Because scoreSessionMatch only credits this terminal's own prompts, a match is always this agent's own conversation, never a neighbour's.

// src/main/claude-fork.ts — resolution order
1. stored id whose session file exists      → resume it            (normal path)
2. id baked into the command whose file exists → adopt it            (legacy forks)
3. match turn history against real files    → recover real session  (the fix)
4. no signal                                   → keep valid id / mint  (new terminal)
Prevention, not just cure: with resolution grounded in what's actually on disk, a minted id can no longer diverge into a phantom — the next reboot resumes correctly on its own.

Recover the seven · runbook

The code fix stops this recurring. These seven agents diverged before the fix and their turn history is frozen, so they're recovered explicitly — safe to run only while the app is down (a live app would flush the phantom ids back over the edit):

  1. Rebuild & hand off to Forge/Conductor

    The running app is the old bundle. Forge rebuilds out/ and coordinates the relaunch — per the ops rule, restarts aren't done from an orch node.

  2. Quit Cookrew

    A clean quit flushes in-memory state to workspace.json; the recovery must happen after that, with the app down.

  3. Arm the bindings

    node recover-sessions.mjs --apply — rewrites the seven claudeSessionId values to their real sessions, backs up workspace.json, and refuses to run if the app is still up.

  4. Relaunch

    claudeSpawnCommand now sees each real file and boots claude --resume <id>. Every agent picks up exactly where the reboot cut it off.
